# Valor Struct Five-Layer AI Organism Architecture

Valor Struct is not an LLM.
Valor Struct is a governed AI organism that orchestrates LLMs, deterministic engineering engines, knowledge graphs, and capability packs through PatchD governance, P-Agent execution, workflow orchestration, and a provider-agnostic AI Gateway.

Valor Struct is a governed engineering AI organism that uses PatchD governance, P-Agent execution, an AI Gateway, and domain capability packs to generate audited engineering, manufacturing, quotation, drawing, and application outputs.

## The five layers

1. PatchD — Governance
2. P-Agent — Execution
3. Workflow Orchestration — Multi-step coordination
4. AI Gateway — Model-role routing
5. Capability Packs — Domain specialists

## 1. PatchD — Governance

PatchD is the governance organ and governance kernel, not an LLM.

It provides:

- stability evaluation
- policy enforcement
- adaptive learning
- meta-governance
- auditability
- reproducibility
- safety guardrails

## 2. P-Agent — Execution

P-Agent is the execution organ: the AI engineer/executor operating under PatchD governance.

It provides:

- task planning
- capability selection
- workflow execution
- memory storage
- report generation
- result packaging

## 3. Workflow Orchestration — Multi-step coordination

Workflow Orchestration Core converts natural requests into deterministic execution chains.

It provides:

- intent classification
- capability selection
- plan building
- workflow execution
- audit bridging
- consolidated report generation

## 4. AI Gateway — Model-role routing

AI Gateway is provider-agnostic. Capabilities request AI roles, not model/provider names.

```text
fast_low_cost_model
policy_reasoning_model
reasoning_model
engineering_reasoning_model
orchestration_model
code_generation_model
local_private_model
fallback_model
```

## Required AI Gateway role mapping

| Organ / Task | AI Gateway Role |
| --- | --- |
| Stability Engine | fast_low_cost_model |
| Governance Policy Engine | policy_reasoning_model |
| Adaptive Policy Engine | reasoning_model |
| Meta-Governance Engine | orchestration_model |
| Task Planner | reasoning_model |
| Intent Classifier | reasoning_model |
| Capability Selector | orchestration_model |
| Workflow Executor | deterministic executor with orchestration_model support |
| Engineering Explanation | engineering_reasoning_model |
| App / API Code Generation | code_generation_model |
| Confidential Project Data | local_private_model |
| Provider Failure / Backup | fallback_model |

Role-based routing:

```text
fast_low_cost_model → Stability evaluation and lightweight classification
policy_reasoning_model → Governance decision support
reasoning_model → Planning and adaptive learning
engineering_reasoning_model → Engineering explanation and review support
orchestration_model → Multi-step workflow reasoning
code_generation_model → Future app/API/code generation
local_private_model → Sensitive or private enterprise workloads
fallback_model → Provider/model fallback route
```

## 5. Capability Packs — Domain specialists

Specialists are governed capability specialists, not fixed AI model names.

Use the term **Governed Capability Specialists**. Do not describe specialists as fixed AI model packs because not every specialist is an LLM.

Specialists = Governed Capability Specialists

Not every specialist is an LLM. Some are deterministic engineering engines.
